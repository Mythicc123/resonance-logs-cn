# Changelog v0.0.4

## Logs

**Buff related**

- Added buff group display, monitor all buffs, and buff priority sorting
- Removed talent-linked buff logic; now manually concatenating buff_name for matching (if you were monitoring talent-type buffs, you will need to reconfigure them)
- Differentiated between user-selected and default monitored items to avoid unnecessary display; only user-selected items are shown
- Reset no longer clears buff-related settings, preventing interruption or loss of buff display continuity

**DPS related**

- Refactored multiple areas to simplify processing
- Refactored overall SQLite operation logic; due to table deletions, you need to delete `resonance-logs-cn.db` from the `%LOCALAPPDATA%\resonance-logs-cn` directory and restart the application
- Added parsing of dungeon info packets; resets are now based on target transitions and wipe buffs. Within the same scenario, resets are delayed (trash mobs cleared -> upon reaching the boss, reset triggers after the first attack on the boss; master dungeon wipe -> reset triggers on the next boss attack)
- History now shows DPS distribution per target. Healing view for individual characters additionally shows healing distribution per party member alongside healing skill breakdown
- Further expanded display of damage ID composition per skill
- Fixed frontend features: hide names, display class specialization, etc.
- Fixed font settings

## Notes

- If you previously used 0.0.2/0.0.3, the first time you use 0.0.4 you need to delete `resonance-logs-cn.db` from the `%LOCALAPPDATA%\resonance-logs-cn` directory and restart the application
- If you were monitoring talent-type buffs, you will need to reconfigure them
